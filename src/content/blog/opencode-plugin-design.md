## 核心设计思想

OpenCode 的插件机制采用了**钩子（Hooks）模式**，这是一种强大而灵活的扩展机制。与传统的基于继承或接口的扩展方式不同，OpenCode 允许插件在关键的生命周期节点切入，进行定制化处理。

## 关键特性

### 1. 非侵入式扩展

插件不需要修改 OpenCode 的核心代码，通过钩子机制就可以实现功能扩展。这样：
- 保持 OpenCode 核心的稳定性
- 允许多个插件共存而不产生冲突
- 便于插件的独立开发和更新

### 2. 渐进式集成

开发者可以从简单的工具扩展开始，逐步深入到 Agent 配置、消息转换等高级功能。

### 3. 完整的生命周期覆盖

从配置初始化、消息处理、工具执行、权限控制等多个环节，都提供了钩子支持。

## 主要钩子分类

### 配置层钩子
- `config`: 修改或读取 OpenCode 配置，包括注册新 Agent

### 工具层钩子  
- `tool`: 注册自定义工具

### 消息处理钩子
- `chat.message`: 收到新消息时触发
- `chat.params`: 修改发送给 LLM 的参数
- `experimental.chat.messages.transform`: 修改消息历史
- `experimental.chat.system.transform`: 修改 System Prompt

### 执行控制钩子
- `command.execute.before`: 命令执行前拦截
- `tool.execute.before/after`: 工具执行前后拦截
- `permission.ask`: 拦截权限请求

## 开发最佳实践

### 1. 清晰的职责划分
- 确保每个钩子只处理单一职责
- 不要在一个钩子中做过多的工作

### 2. 错误处理
- 提供完善的错误处理机制
- 避免插件异常导致 OpenCode 崩溃

### 3. 性能考虑
- 避免在热点路径（如 message 钩子）做过重的计算
- 使用缓存减少重复计算

### 4. 文档完善
- 为自定义工具和 Agent 提供详细的说明
- 说明插件的权限需求

## 总结

OpenCode 的插件机制通过钩子模式实现了一个优雅、可扩展的架构。它既保证了系统的稳定性，又为开发者提供了强大的定制能力。这对于构建可扩展的 AI 编程工具具有重要的参考价值。
